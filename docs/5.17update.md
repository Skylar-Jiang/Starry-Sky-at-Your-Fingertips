一、重做“捏云朵”体验
--------------------------------

当前问题：
云朵只能微微移动，看起来不像“捏”，更像一个模糊贴图。

目标：
做成一个可明显互动的“云朵玩具”：
- 至少 3 朵云。
- 每朵云都能被鼠标/触屏拖动。
- 拖动范围覆盖主星空区域的大部分。
- 可以明显被压扁、拉伸、揉动。
- 揉到一定程度后散成星尘。
- 焦虑情绪是“揉散乱云”。
- 平静情绪是“捏出云星”。

建议新增或重构：
- src/components/CloudKneadInteraction.jsx
- src/hooks/useCloudKneadController.js
- src/utils/gestureUtils.js
- src/config/microInteractionConfig.js 中 cloudKnead 配置

多云设计：
- active 时生成 3 到 5 朵云。
- 云朵初始位置分散，例如：
  - 20% x, 32% y
  - 50% x, 42% y
  - 72% x, 30% y
  - 36% x, 62% y
- 每朵云大小略有差异。
- 每朵云有自己的状态：
  - idle
  - grabbed
  - kneading
  - dissolved
- 每朵云有自己的 kneadProgress，0 到 1。
- 完成任意 2 朵或 3 朵后，可触发 onComplete，按体验选择。

鼠标/触屏交互：
- pointerdown 在云朵上时：
  - 选中该云。
  - setPointerCapture。
  - 记录 pointer 起点。
  - 云朵进入 grabbed。
- pointermove 时：
  - 云朵位置跟随 pointer，而不是只做小幅 transform。
  - 限制在容器范围内，不能飞出屏幕。
  - 根据移动距离累积 kneadProgress。
  - 根据拖动速度和方向做 squash/stretch：
    - 水平拖动明显时 scaleX 1.08-1.18，scaleY 0.88-0.96。
    - 垂直拖动明显时 scaleY 1.06-1.14，scaleX 0.9-0.98。
    - 轻微 rotate -6deg 到 6deg。
  - 在云朵两侧叠加柔光压力点或手指压痕。
- pointerup 时：
  - releasePointerCapture。
  - 如果 kneadProgress 未满，云朵轻轻回弹到 idle 或半 squeezed。
  - 如果 kneadProgress 满，进入 dissolved，散成星尘。

“捏”的视觉必须明显：
- 拖动时不要只是 translate。
- 必须叠加：
  1. squash/stretch 形变。
  2. squeezed 图片或 squeezed CSS 状态。
  3. 云朵内部亮度变化。
  4. 星尘粒子。
  5. 进度反馈，例如云朵底部的小星点进度或云体变薄。
- 若已有 cloud_idle/cloud_squeezed/cloud_dissolve 图片，继续使用。
- 如果图片效果还是像模糊雾，请增强边缘层次：
  - 加一层柔光轮廓。
  - 加少量内部星点。
  - 加轻微蓝紫渐变。
  - 不要只靠 blur。

可操作范围：
- 云朵容器应覆盖主星空可视区域，例如 absolute inset: 0。
- 云朵自身 pointer-events: auto。
- 非交互背景 pointer-events: none。
- 不要把云朵限制在一个很小的子容器中。
- 不要让父层 overflow hidden 裁掉主要拖动范围，除非是全屏边界。

二、加入手势 pinch 操作云朵
--------------------------------

目标：
如果摄像头和 hand landmarks 可用，用户用拇指和食指捏合可以抓住云朵并拖动。

不要强依赖 GestureRecognizer 内置类别。
优先用 HandLandmarker landmarks 自己算 pinch。

pinch 检测建议：
- 使用同一只手的 thumb tip 和 index finger tip：
  - thumbTip = landmark[4]
  - indexTip = landmark[8]
- 计算二者距离 pinchDistance。
- 用手掌大小归一化，例如：
  - palmSize = distance(landmark[0], landmark[9]) 或 distance(landmark[5], landmark[17])
  - normalizedPinch = pinchDistance / palmSize
- 阈值建议：
  - pinchStart: normalizedPinch < 0.35
  - pinchEnd: normalizedPinch > 0.48
- 必须使用 hysteresis，避免抖动。
- pinch 点位置：
  - midpoint(thumbTip, indexTip)
  - 从 video 坐标映射到屏幕/容器坐标。
- 使用 smoothing：
  - smoothedX = previousX * 0.75 + currentX * 0.25
  - smoothedY = previousY * 0.75 + currentY * 0.25

pinch 状态机：
- idle：
  - 检测 pinchStart。
  - 找最近云朵，距离小于 pinchSelectRadius 才选中。
- pinching：
  - 移动选中的云朵到 pinch 点附近。
  - 累积 kneadProgress。
  - 云朵进入 squeezed/kneading 状态。
- released：
  - 检测 pinchEnd。
  - 释放云朵。
  - 若进度满则 dissolve。

必须有可见反馈：
- 当检测到 pinch 但没选中云朵时，显示一个小光圈跟随手指。
- 当选中云朵时，云朵边缘发光。
- 面板或小提示显示：
  - “捏合手指来抓住云朵”
  - “已抓住云朵”
  - “松开手指让云朵散开”

如果摄像头不可用：
- 鼠标/触屏照样可用。
- 不报错。
- 显示“可用鼠标拖动云朵”。

不要只依赖内置 GestureRecognizer category，因为合掌 / namaste 很可能不是内置类别。
请基于两只手的 landmarks 写启发式检测。

双手合掌检测算法建议：

输入：
- hands: MediaPipe 检测到的 hand landmarks 数组。
- 每只手 21 个 landmarks。
- 如果有 handedness，可以区分 left/right；没有也可以用 x 坐标排序。

必要条件：
1. 同时检测到两只手。
2. 两只手掌中心距离足够近。
3. 两只手高度接近。
4. 两只手大致竖直。
5. 手指方向大致向上或接近竖直。
6. 条件连续稳定 800ms 到 1200ms 后才触发。

关键点：
- wrist = landmark[0]
- indexMcp = landmark[5]
- middleMcp = landmark[9]
- ringMcp = landmark[13]
- pinkyMcp = landmark[17]
- indexTip = landmark[8]
- middleTip = landmark[12]
- ringTip = landmark[16]
- pinkyTip = landmark[20]
- thumbTip = landmark[4]

palmCenter:
- average of wrist, indexMcp, middleMcp, ringMcp, pinkyMcp

handScale:
- distance(wrist, middleMcp) 或 distance(indexMcp, pinkyMcp)

判断 palm centers close：
- distance(leftPalmCenter, rightPalmCenter) / averageHandScale < 1.3
- 可根据实际调试调整到 1.5 或 1.8，不要太苛刻。

判断高度接近：
- abs(leftPalmCenter.y - rightPalmCenter.y) / averageHandScale < 0.7

判断竖直：
- 对每只手，middleTip.y < wrist.y 或 middleTip.y < middleMcp.y
- 注意 video 坐标 y 向下，tip.y 更小代表更靠上。

判断双手相对靠近：
- leftThumbTip 和 rightThumbTip 或 leftIndexMcp 和 rightIndexMcp 距离较小。
- 但不要只用 thumb tips，因为合掌时拇指可能被遮挡。

稳定触发：
- 每帧计算 prayerScore，0 到 1。
- 连续 prayerScore > 0.75 达到 holdMs = 1000ms，触发一次。
- 触发后 cooldown 3000ms，避免重复触发。
- 如果中途分开，进度回落但不要立刻清零，可做 200ms 容错。

UI 状态反馈：
在手势面板或主界面小提示显示：
- 没有摄像头：“打开摄像头后，可以合掌许愿。”
- 没有双手：“正在寻找双手…”
- 检测到单手：“再伸出另一只手，就可以合掌许愿。”
- 检测到双手：“双手已识别，请慢慢合掌。”
- 接近合掌：“保持合掌 40%”
- 成功：“合掌许愿成功。”
- 太远：“双手再靠近一点。”

进度展示：
- 可以用一个小圆环、进度条或文字百分比。
- 这比静默失败重要。

接入要求：
- triggerWishRitual("gesture") 必须能被真实 prayer detection 调用。
- 保留“模拟合掌许愿”按钮，用于测试和无摄像头环境。
- 保留普通点击/长按许愿按钮。
- 摄像头权限失败时不能影响页面其他功能。

调试要求：
开发环境中可以显示 gesture debug 信息：
- handsCount
- prayerScore
- prayerHoldProgress
- pinchDetected
- selectedCloudId

生产环境可以隐藏或折叠。

四、修复流星雨为可重复触发
--------------------------------

当前问题：
流星雨范围和时间短，而且像只能使用一次，需要刷新。

目标：
- 每次触发都播放。
- 不需要刷新。
- 可以连续触发，但有合理 cooldown。
- 范围覆盖更多屏幕。
- 时间更长，更像流星雨。

重构建议：
不要只用 active boolean 控制。
改用 triggerKey / eventId / sequenceId。

示例思路：
- 父组件维护 meteorEventId。
- 每次触发：
  - setMeteorEventId(id => id + 1)
- MeteorShowerLayer 接收 triggerKey。
- triggerKey 变化时重新生成 meteor descriptors 并播放。
- 播放结束后内部清理，但不要阻止下一次 triggerKey 播放。

props 建议：
MeteorShowerLayer({
  triggerKey,
  durationMs = 6500,
  count = 24,
  reducedMotionCount = 4,
  onComplete
})

流星雨增强：
- 持续时间：5 到 8 秒。
- 数量：普通 18 到 28 条。
- reduced-motion：3 到 5 条。
- 出现范围：
  - 起点分布在屏幕上方和右侧。
  - 终点向左下方移动。
  - 不要只集中在一小块区域。
- delay 分布在整个 duration 内。
- 每条流星长度、速度、透明度随机但稳定。
- 使用 useMemo 基于 triggerKey 生成，不要每帧变化。
- pointer-events: none。

视觉要求：
- 流星轨迹更长，但不能像激光。
- 透明度更柔和。
- 可以有少量短流星和 2 到 3 条明显长流星。
- 不要挡住弹窗文字太久。

状态修复：
- onAnimationEnd 或 setTimeout 完成后清理当前 descriptors。
- 如果中途再次触发，旧动画可以自然结束，也可以替换为新一场。
- 不要留下 active 永远 true 的状态。
- 不要让 CSS animation name 复用导致浏览器不重播。

增加按钮：
- 如果界面适合，可以加一个小按钮“再看一场流星雨”或“唤来流星雨”。
